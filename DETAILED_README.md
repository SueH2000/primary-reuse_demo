This document is the detailed project record for the current repository state.
It explains:

- what problem we are solving
- what code was added and why
- how RAG is integrated
- exactly how the `~200` labeled papers are used
- what the current benchmark results do and do not prove

---

## 1. Problem Definition

The task is to classify a biomedical paper into one of these labels:

- `Primary`: the paper mainly uses data generated by the authors themselves
- `Reuse`: the paper mainly reuses public or external datasets
- `Unclear`: the evidence is insufficient or contradictory

In this project, the most important practical observation was:

- reading the full paper is often too noisy
- the best signal usually appears near accession IDs and provenance phrases
- focused evidence extraction is therefore more useful than raw full-text input

That observation drove the current design.

## 3. What We Built

The repository now has two distinct but related pipelines.

### A. `rag_lightweight_pipeline.py`

Purpose:

- practical inference pipeline
- retrieves similar labeled examples
- optionally sends the target evidence plus retrieved examples to a local LLM

Use this when you want:

- paper-by-paper predictions
- prompt-based RAG
- a reproducible inference workflow

### B. `evidence_modeling.py`

Purpose:

- fair offline benchmark on the gold-standard labeled set
- compares multiple methods on the same split or across cross-validation folds

Use this when you want:

- model comparison
- extraction comparison
- evidence-window sensitivity analysis
- audit-ready evaluation artifacts

The benchmark script currently compares:

- `static_rules`
- `mined_template_rules`
- `linear_model`
- `linear_model_plus_rag`
- `rag_vote`
- `rules_plus_rag`
- `rag_llm` (optional)

In this repository, the standard textbook RAG path is:

- retrieve labeled support examples from the training fold
- send the target evidence plus those retrieved examples to the LLM
- let the LLM classify directly

That corresponds to:

- `rag_llm` with `--llm-strategy classify`

The repository now also supports production-style routing:

- estimate confidence from `linear_model_plus_rag`
- route each paper to `auto_accept`, `llm_review`, or `human_review`
- optionally call the LLM only on the `llm_review` subset with `--llm-eval-mode production_route`

## 4. Why We Did Not Use Full Text Directly

The project moved away from naive full-text classification because the benchmark showed that focused evidence works better.

Current evidence sensitivity result:

- `title_only + linear_model`: `0.684`
- `full_text_12000 + linear_model`: `0.789`
- `accession_default + linear_model`: `0.854`
- `accession_default + rag_vote`: `0.819`

Interpretation:

- title only loses too much information
- long full text adds noise
- focused provenance windows preserve the strongest clues

This is why the current default extractor is:

- `350` characters before each accession/provenance hit
- `900` characters after
- `2200` maximum evidence characters per article

## 5. What "RAG" Means in This Project

RAG here means:

- retrieve similar labeled examples
- add them as support context for the target paper
- then either:
  - predict directly from retrieved labels (`rag_vote`), or
  - pass the evidence plus retrieved examples to an LLM (`rag_llm`)

It does **not** mean:

- updating model weights
- fine-tuning on the 200 labels
- training a transformer from scratch

That distinction matters.

### In this project, RAG is retrieval-time supervision, not weight training

The 200 labeled papers are used as a local example bank.

They teach the system in two ways:

1. As retrieval examples
   - similar labeled papers are shown to the model or used for weighted voting

2. As benchmark supervision
   - they let us measure whether a method is actually improving

They do **not** currently teach the model through LoRA or full fine-tuning.

## 6. How the 200 Labeled Papers Are Used in `rag_lightweight_pipeline.py`

This is the prompt-based RAG pipeline.

### Step 1: Load labeled examples

The script reads the labeled CSV and keeps only rows labeled `Primary` or `Reuse`.

It builds one retrieval text per labeled paper from:

- `title`
- `abstract`
- `full_text`

That retrieval text is used only for similarity search.

### Step 2: Downsample to a manageable local retrieval bank

The script does not dump all 200 labeled examples into the prompt.

Instead it:

- keeps labels balanced
- samples up to `rag_max_examples` total
- default `rag_max_examples = 30`

Why:

- prompt size must stay bounded
- balanced sampling reduces bias toward the larger class
- fast iteration is more important than building a heavy retrieval stack at this stage

Important detail:

- this prompt-time pipeline is optimized for practical local use, not for strict benchmark fairness

### Step 3: Build a TF-IDF retriever

The script vectorizes the local labeled bank using:

- TF-IDF
- unigram and bigram features
- English stop-word filtering

Then it computes similarity with:

- cosine similarity

Why this simple retriever was chosen:

- no vector database required
- no embedding API dependency
- works well enough for a small labeled dataset
- easy to debug

### Step 4: Build the target paper evidence

For each target article, the script first extracts focused evidence from the JSONL article text.

Default extraction:

- accession/provenance windows
- fallback to provenance keywords if no accession window is found

This gives one short evidence bundle per target paper.

### Step 5: Retrieve top-k labeled neighbors

The target evidence is used as the retrieval query.

Default retrieval settings:

- `rag_top_k = 3`
- `rag_candidate_pool = 12`
- `rag_per_label_cap = 2`
- `rag_min_similarity = 0.05`

Interpretation:

- search a slightly larger candidate set
- keep only sufficiently similar rows
- cap how many examples can come from one label
- return a small final set to keep the prompt readable

### Step 6: Inject retrieved examples into the prompt

The retrieved examples are inserted into the LLM prompt as short labeled support examples.

The prompt also includes:

- detected accessions
- extracted evidence text
- optional phrase hints mined from the benchmark script

So the LLM sees:

1. the target evidence
2. similar labeled papers
3. class-associated phrases
4. explicit decision rules

### Step 7: Predict label

The practical pipeline can operate in different modes:

- heuristic only
- heuristic then LLM on unclear cases
- LLM for all rows

If the LLM is enabled, it returns:

- `label`
- `confidence`
- `rationale`

## 7. How the 200 Labeled Papers Are Used in `evidence_modeling.py`

This is the benchmark pipeline.

This is the script you should trust for fair performance comparison.

### Step 1: Join labels with article text

The script:

- reads the labeled CSV
- normalizes labels
- keeps only `Primary` and `Reuse`
- joins with the local JSONL by `paper_id`

In the current local run:

- `200` labeled rows exist
- `199` rows are usable after the join

### Step 2: Extract evidence from matched article text

The benchmark does **not** retrieve using raw full text by default.

It first transforms each article into a compact `evidence_text`.

That evidence is what the benchmark models consume.

Why:

- this makes the comparison about provenance signal, not raw document length
- retrieval becomes more focused
- classical models also benefit from less noise

### Step 3: Split into train and test

For one held-out benchmark split:

- about `80%` of the matched labeled rows go to train
- about `20%` go to test

With `199` matched rows, that is roughly:

- `159` training rows
- `40` test rows

This matters for RAG:

- only training rows are allowed in the retrieval bank for that split
- test rows are never allowed to retrieve themselves or each other

That is the key anti-leakage rule.

The repository now also supports saving and reusing an explicit split manifest:

- `heldout_split.json`

This matters because:

- a fixed random seed is helpful but still indirect
- an explicit split file guarantees that every compared run uses the same paper IDs in train and test

### Step 4: Build train-fold-only retrieval bank

For `rag_vote` and `rag_llm`, the benchmark retriever is built only from:

- training fold `title`
- training fold `evidence_text`
- training fold `label`

The benchmark retriever uses:

- TF-IDF
- cosine similarity
- top-k retrieval

The newer provenance-aware version keeps that simple backbone, but changes the scoring:

- each paper is compressed into provenance-dense spans instead of broad article text
- retrieval text is augmented with explicit cue tokens such as `cue_reuse`, `cue_primary`, `cue_deposit`, and `cue_download`
- neighbor scores are adjusted upward when the target and candidate share the same provenance cue pattern
- neighbor scores are adjusted downward when the target looks like `Reuse` but the candidate only shows `Primary` cues, or vice versa
- extracted spans are also assigned explicit roles such as `Primary_generation`, `Primary_deposition`, `Reuse_download`, and `Reuse_reanalysis`

This keeps the retriever auditable while making it less dependent on broad topic similarity.

### Step 5: Predict by weighted vote or by LLM

There are two RAG-style benchmark paths.

#### A. `rag_vote`

The system retrieves nearest labeled training examples and then:

- sums similarity scores by label
- predicts the label with the strongest total similarity

This is the cleanest retrieval-only baseline.

It answers:

- how much performance do we get from retrieval alone, without an LLM

#### A2. `linear_model_plus_rag`

This hybrid model keeps the original TF-IDF evidence classifier, then appends RAG-derived features such as:

- `Primary` neighbor score
- `Reuse` neighbor score
- vote margin
- neighbor count
- provenance cue counts from the target evidence
- structured role counts from the target evidence

Why this matters:

- the linear classifier is good at local lexical cues
- RAG is good at giving structured support about which label similar training papers received
- combining them is often safer than using retrieval vote alone

#### B. `rag_llm`

The system retrieves nearest labeled training examples and then sends:

- target accessions
- target evidence
- retrieved neighbors
- mined phrase hints

to the LLM.

This answers:

- whether an LLM can use retrieved evidence better than weighted voting alone

There are now two practical styles for `rag_llm` evaluation:

1. Direct classification
   - the LLM predicts the label for every called row
   - this is the standard textbook RAG baseline

2. Verifier / override mode
   - `rag_vote` makes the first prediction
   - the LLM reviews only harder rows
   - the LLM decides whether to keep or override the RAG vote

The second style is currently safer for this task.

### Step 6: Save audit artifacts

The benchmark saves:

- overall comparison CSVs
- row-level predictions
- retrieved neighbors
- `standard_rag_preview.csv`, which shows the exact retrieve-then-classify context for the standard RAG baseline
- `route_summary.csv`, which summarizes how many rows would be auto-accepted, escalated to LLM review, or sent to human review
- fold metrics
- evidence sensitivity outputs
- LLM-call coverage summary when applicable

This makes the benchmark inspectable instead of opaque.

## 8. Why the Benchmark Uses Train-Fold-Only RAG

If all 199 matched labeled papers were always available for retrieval, benchmark accuracy would be inflated.

That would be leakage.

The benchmark prevents this by:

- splitting first
- building the retriever only from the training subset
- evaluating on held-out rows

This is the most important fairness rule in the current codebase.

Without it, a high RAG score would not be trustworthy.

## 9. What the Current Results Mean

### Current deterministic benchmark result

Held-out split:

- `static_rules`: `0.375`
- `mined_template_rules`: `0.500`
- `linear_model`: `0.725`
- `rag_vote`: `0.800`
- `rules_plus_rag`: `0.700`

Interpretation:

- retrieval is clearly useful
- simple retrieval-only voting currently beats the heuristic baseline
- the strongest single held-out split result is `rag_vote`

Updated provenance-aware fixed-split result:

- `linear_model`: `0.725`
- `linear_model_plus_rag`: `0.775`
- `rag_vote`: `0.750`

Interpretation:

- provenance-aware retrieval improved the hybrid classifier
- the same retrieval logic did not improve raw `rag_vote` enough to become the best standalone method
- in this dataset, RAG currently works better as extra evidence features than as the final classifier

Important refinement from the structured-role rerun:

- `accession_short + rag_vote`: `0.854`

Interpretation:

- when RAG is forced to look only at short, tightly focused provenance windows, the structured role layer helps much more
- broader windows still introduce enough mixed context that raw retrieval vote becomes less reliable

### Current more reliable result

5-fold cross-validation:

- `accession_windows + linear_model`: `0.854`
- `accession_windows + linear_model_plus_rag`: `0.850`
- `section_aware + linear_model_plus_rag`: `0.829`
- `section_aware + linear_model`: `0.824`
- `accession_windows + rag_vote`: `0.779`
- `section_aware + rag_vote`: `0.658`

Interpretation:

- on average across folds, the linear model is currently the strongest validated method
- provenance-aware RAG helps more when used as hybrid features than when used as raw vote
- RAG still helps, but it is not automatically better than a strong classical baseline
- this is not a contradiction; it means retrieval alone is helpful, but not yet dominant across every split

## 10. What the Current LLM+RAG Results Do and Do Not Prove

Right now, this repository does **not** yet prove that `rag_llm` improves over `rag_vote`.

Reason:

- one historical full run failed because of output-parsing issues
- one later smoke run only called the LLM on `5/40` rows and fell back to `rag_vote` on the other rows

So:

- we have infrastructure for `rag_llm`
- we have prompt integration
- we have evaluation hooks
- but we do not yet have a clean full-run result that proves improvement

That is why the benchmark now records:

- `pred_rag_llm_called`
- `llm_evaluation_summary.csv`

This prevents ambiguous claims.

Updated status:

- the old direct full-row `rag_llm` run was clearly worse than `rag_vote`
- after switching to a reviewer-style `verify_override` strategy on `hard_cases`, a new held-out split run reached:
  - `rag_vote = 0.775`
  - `rag_llm = 0.800`

Interpretation:

- this is a real improvement on that split
- the improvement is small and comes from selective overrides, not from broad LLM dominance
- this is encouraging, but still not enough to claim that LLM review is universally better than retrieval-only voting

## 11. Why RAG Did Not Automatically Beat the Linear Model

This is an important project lesson.

A strong linear model can outperform a prompt-based LLM+RAG setup when:

- the labeled set is still small
- the retrieval examples are noisy or mixed
- the prompt is not strong enough
- the task depends on a few local textual cues that TF-IDF already captures well

That is exactly why this repository separates:

- extraction quality
- retrieval quality
- LLM prompting quality
- fair evaluation

RAG is not magic by itself.

The retrieval must be relevant, and the LLM must use it correctly.

## 14. Recommended Next Technical Steps

If the goal is to make `RAG+LLM` clearly better, the next steps should be:

1. Get a clean full LLM run with no endpoint interruptions.
2. Compare `rag_vote` vs `rag_llm` only on rows where the LLM was actually called.
3. Improve retrieval quality before trying LoRA.
4. Add more hard negative examples to the labeled set.
5. Only consider LoRA after the extraction pipeline and benchmark protocol are stable.

If the goal is to maximize near-term benchmark accuracy, the current best direction is:

- keep improving evidence extraction
- keep the benchmark strict
- treat the linear model as a serious baseline, not as a placeholder
