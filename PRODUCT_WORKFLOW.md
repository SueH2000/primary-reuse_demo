# Product Workflow

This document defines the deployable workflow for the biomedical paper
classification service.

It is written as a product contract rather than as an experiment log.

---

## 1. Product Goal

The system should help a curator or analyst decide whether a biomedical paper:

- `Primary`: mainly uses data generated by the authors
- `Reuse`: mainly reuses public or external datasets
- `Unclear`: evidence is mixed or insufficient

The system is not only a label predictor.
It is an evidence-backed classification assistant.

---

## 2. Main User Flows

### A. Paste article text

User input:

- `title` optional
- `text` required

System action:

- extract evidence windows
- extract accession IDs and `GSE`
- classify with `linear_model_plus_rag`
- optionally call LLM if routed to review

System output:

- predicted label
- confidence
- route recommendation
- evidence spans
- main decision sentence
- extracted `GSE`
- direct `GSE` links
- retrieved support examples
- optional side-by-side comparison across hybrid baseline, LLM sentence judge, and LLM final classify

### B. Lookup by local identifier

User input:

- `paper_id`
- or other locally indexed identifier

System action:

- resolve the local article text
- run the same classification pipeline

System output:

- same as paste-text flow

Current limitation:

- local lookup still gives the fastest path
- local Mohammad DOI -> PMCID mapping is now available as a first DOI bridge when the mapping file exists
- DOI and PMCID can now fall through to a cached Europe PMC resolver when not found in the local index
- remote DOI resolution still depends on Europe PMC availability and article coverage

### C. Batch upload

User input:

- `.csv`
- `.jsonl`
- `.json`

Accepted row fields:

- `text`
- `full_text`
- `article_text`
- `title`
- `paper_id`
- `pmcid`
- `doi`
- `identifier`

Row processing order:

1. If row contains article text, classify from that text directly
2. Else if row contains identifier, try local lookup
3. Else return an error for that row

System output:

- per-row result list
- count
- full prediction payload for each row
- or, in async mode, a job ID plus downloadable JSON/CSV artifacts

### D. One-click Mohammad GEO/GSE subset import

User input:

- optional sample size
- optional random seed
- optional output JSONL filename

System action:

- load the default Mohammad DOI/GEO mapping CSV
- keep only GEO/GSE-linked records
- prefer already indexed local article text
- fall back to resolver/cache only for missing PMCIDs
- write a reusable JSONL artifact for later training/classification

System output:

- import job ID
- import job status
- downloadable JSONL artifact
- optional activation of that JSONL as the current extra lookup source

### E. Feedback correction

User input:

- predicted label
- corrected label
- optional reviewer
- optional note
- optional text or identifier

System action:

- resolve article if possible
- rerun core evidence extraction for audit consistency
- append feedback record to feedback gold-standard CSV

System output:

- saved status
- feedback file path
- stored record

### F. Refresh reviewed RAG bank

User input:

- optional output CSV path
- optional report JSON path
- optional `activate=true/false`

System action:

- read `manual_ground_truth_with_GSE_links_REFRESHED.csv`
- read `rag_feedback_gold_standard.csv`
- merge only human-reviewed rows into a refreshed labeled bank
- write a refreshed CSV and audit report
- optionally activate the refreshed labeled bank for future inference

System output:

- refreshed CSV path
- report JSON path
- updated row count
- appended row count
- activation status

## 4. Output Contract

Core output fields:

- `final.label`
- `final.source`
- `recommended_route`
- `recommended_route_reason`
- `predictions.linear_model_plus_rag`
- `predictions.linear_model_plus_rag_conf`
- `predictions.rag_vote`
- `evidence.gse_ids`
- `evidence.gse_urls`
- `evidence.accession_list`
- `evidence.main_decision_sentence`
- `evidence.main_decision_role`
- `evidence.structured_evidence_summary`
- `rag.neighbors`

Example:

```json
{
  "final": {
    "label": "Reuse",
    "source": "linear_model_plus_rag"
  },
  "recommended_route": "llm_review",
  "recommended_route_reason": "rag_disagree|low_rag_margin",
  "predictions": {
    "linear_model_plus_rag": "Reuse",
    "linear_model_plus_rag_conf": 0.81,
    "rag_vote": "Primary"
  },
  "evidence": {
    "gse_ids": ["GSE12345"],
    "gse_urls": ["https://www.ncbi.nlm.nih.gov/geo/query/acc.cgi?acc=GSE12345"],
    "accession_list": ["GSE12345"],
    "main_decision_sentence": "We downloaded GSE12345 from GEO and reanalyzed it.",
    "main_decision_role": "Reuse_download",
    "structured_evidence_summary": "[Reuse_download] ..."
  },
  "rag": {
    "neighbors": []
  }
}
```

## 5. Where the Current System Is Still Rough

### DOI lookup

Current state:

- local identifier lookup is dependable
- DOI has a partial ingestion path through Mohammad mapping and the resolver/cache layer

Needed next:

- broader DOI coverage beyond Europe PMC / local mapping
- additional publisher/full-text fallbacks
- stronger diagnostics for remote fetch failures

### Feedback loop

Current state:

- feedback is appended to a separate CSV
- safer than overwriting the original gold standard

Available now:

- merge command to produce a new labeled CSV from base labels plus reviewer feedback
- refresh-and-activate API/UI flow to promote reviewed feedback into the active labeled/RAG bank

Needed next:

- review queue UI
- optional scheduled retraining command using original labels + approved feedback

### Frontend

Current state:

- backend API exists
- Swagger docs exist
- local operator UI exists at `/`

Available now:

- paste-text review page
- local identifier lookup page
- batch upload page
- feedback form wired to the backend
- evidence and `GSE` comparison panel

Needed next:

- DOI-first ingestion page
- asynchronous batch job progress for large uploads
- reviewer queue and adjudication workflow

### Batch operations

Current state:

- synchronous upload still exists for small files
- async batch job flow now exists with persisted status and downloadable result files

Needed next:

- richer progress updates beyond queued/running/completed
- cancel/retry controls

## 6. Recommended MVP Product Shape

### Backend

- FastAPI service
- local Ollama optional

### Local UI

- mounted at `/`
- browser-based operator console
- built on the same API contract as programmatic clients
- file-based feedback store

### Frontend

Three tabs:

1. `Single Paper`
   - paste title + text
   - or paste identifier

2. `Batch Upload`
   - upload CSV / JSONL
   - preview columns
   - run batch classification

3. `Review Queue`
   - show routed `llm_review` and `human_review` papers
   - show label, confidence, `GSE`, evidence, and correction controls

### Feedback data lifecycle

1. user corrects output
2. feedback appended to `rag_feedback_gold_standard.csv`
3. curator approves merge
4. updated training set is rebuilt
5. production classifier is retrained
